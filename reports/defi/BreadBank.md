### [HIGH-01] Unchecked Transfer Calls

**Severity:** HIGH  
**Location:** Lines 20-22, 34-37  
**CVE:** N/A

**Description:**  
The contract uses `.transfer()` and `.send()` without verifying return values. In Solidity, these functions can fail silently, meaning funds could be stuck without the caller knowing.

**Code Snippet:**
```solidity
owner.transfer(amount); // Could fail silently
```

**Recommendation:**
```solidity
require(owner.transfer(amount), "Transfer failed");
// Or better: use pull over push pattern
```

**Risk:** Failed transfers result in lost funds that cannot be recovered.

### [HIGH-02] Reentrancy Vulnerability

**Severity:** HIGH  
**Location:** Function `withdraw()`  
**CVE:** N/A

**Description:**  
The withdrawal function sends ETH to the user BEFORE updating the internal state (user balance). This is a classic reentrancy pattern where an attacker could recursively call `withdraw()` before the balance is updated.

**Code Snippet:**
```solidity
function withdraw() public {
    uint256 amount = balances[msg.sender];
    (bool sent,) = msg.sender.call{value: amount}("");
    require(sent, "Failed to send Ether");
    balances[msg.sender] = 0; // State updated AFTER call
}
```

**Recommendation:**
```solidity
// Use Checks-Effects-Interactions pattern:
function withdraw() public nonReentrant {
    uint256 amount = balances[msg.sender];
    require(amount > 0, "No balance");
    
    balances[msg.sender] = 0; // Update state FIRST
    
    (bool sent,) = msg.sender.call{value: amount}("");
    require(sent, "Failed to send Ether");
}
```

**Risk:** Attacker could drain all contract funds through reentrancy.

### [HIGH-03] Missing Input Validation

**Severity:** HIGH  
**Location:** `deposit()` function  
**CVE:** N/A

**Description:**  
The deposit function lacks sufficient validation. While `msg.value > 0` is checked, there are no limits on deposit size, which could lead to:
- Integer overflow in balance tracking (pre-Solidity 0.8.0)
- Denial of service if contract runs out of gas

**Recommendation:**
```solidity
function deposit() public payable {
    require(msg.value > 0, "Must deposit ETH");
    require(msg.value <= 1000 ether, "Deposit too large");
    balances[msg.sender] += msg.value;
}
```

## Conclusion

The BreadBank contract has fundamental security flaws that make it unsafe for production use. The reentrancy vulnerability is the most critical issue and could lead to complete loss of funds.

**Recommended Actions:**
1. Implement nonReentrant modifier (OpenZeppelin)
2. Apply Checks-Effects-Interactions pattern
3. Add input validation and limits
4. Emit events for all state changes
5. Add emergency stop functionality
